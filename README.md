# Career Pairwise Lab（キャリア比較ラボ）

2択比較を通じて、キャリアの関心・判断軸・避けたい要素を整理するスマホ向けWebアプリです。適職を断定する診断ではなく、次に調べることや相談時の論点を作るためのツールです。

## 使い方

`index.html` をブラウザで開くだけで利用できます。外部API・外部ライブラリ・ビルド工程は不要で、GitHub Pagesにもそのまま配置できます。

## 主な機能

- 現在地とキャリア形態を選び、会社員・組織所属／個人活動のフローへ分岐
- 興味軸、業界・職種または活動ジャンル・収益モデル、仕事スタイルを2択比較
- 任意の比較理由と、避けたい要素・リスクを記録
- 結果を仮説としてランキング表示し、AI分析用プロンプトを生成・コピー
- 結果をブラウザの `localStorage` に最大30件保存し、履歴から再表示

入力内容は外部へ送信されません。保存データは利用中のブラウザ内にのみ保持されます。

## v0.4.1 の改善点

- AI分析用プロンプトと `localStorage` の履歴に、比較統計とタイブレーク結果を保存

## v0.5 の改善点

- 候補を早期除外せず、代表比較・探索比較・接戦比較・反証比較を組み合わせる探索型比較エンジンへ変更
- 各レイヤーを最大8候補・12問に抑え、傾向適合60%・カテゴリ分散25%・探索15%を目安に出題
- 候補ランキングとは別に16種類のtraitスコアを保持し、内部傾向を推定
- 「当たり前品質」と「魅力的品質」のチェック項目をコードと保存履歴に追加
- 会社員・個人活動・創作・副業を横断する可能性マップを結果画面に追加
- trait相性は高いが比較が少ない「未探索の可能性」を表示
- 結果を鵜呑みにしないための反証質問と、次に試す小さな行動を追加

選択したキャリア形態は比較フローの分類に使いますが、候補自体の存在を消す条件には使いません。結果画面では選択モード外の可能性も参考候補として確認できます。

- ウォームアップ3問、探索8問、仕事スタイル5問、接戦・反証4問の時間予算型フローへ変更

候補マスタの広さは維持し、実際の質問だけを情報量と時間予算に基づいて選びます。途中結果には、回答数が少ない仮説である旨を明示します。

### v0.9.4 動作復旧・バグ修正

`desired_career_modes` 対応の直後、質問に1問回答した時点でアプリが応答不能になる不具合が発生していました。原因と対応は以下の通りです。

- **原因**: 過去のタイブレーク機構の残骸である `S.tieLogs` が、状態初期化コード（`S={...}`／`reset()`）から削除された後も `headToHead()` と `reasonsFor()` の中で `[...S.logs,...S.tieLogs]` として参照され続けていた。`S.tieLogs` が `undefined` のためスプレッド構文が例外を送出し、最初の回答直後に呼ばれる `recalculateAllScores() → ranking() → reasonsFor()` の経路で必ずクラッシュしていた。
- **対応**: 現行アーキテクチャでは全回答ログが `S.logs` に一本化されているため、`[...S.logs,...S.tieLogs]` を `S.logs` に修正（`tieLogs` 自体は廃止済みの概念のため状態への再追加はしていない）。
- **副次修正**: `history()` が保存データ内の `null` や必須フィールド欠落レコードをフィルタせずそのまま返しており、壊れた `localStorage` 履歴が1件でも混ざると履歴画面が丸ごとクラッシュする経路があった。`history()` で `id` を持つオブジェクトのみを残すようフィルタを追加し、`showHistory()` 側の `user_status` 参照もオプショナルチェーンで防御化。
- **互換性**: `desiredModesOf()` は `desired_career_modes`（新形式）と旧 `career_mode`（文字列／オブジェクト）の両方を読み替えるため、v0.6〜v0.9のどの履歴形式でも結果画面・履歴一覧・がっつり版への引き継ぎが破綻しないことを確認済み。
- **確認方法**: `node --check` によるスクリプト構文チェック、および実ブラウザでの標準版・がっつり版それぞれの単独完走、標準版→がっつり版引き継ぎ、戻る操作、旧形式・破損データ混在の履歴表示を一通り動作確認。

## v0.9.5 の改善点（安定化・設計整合・戻る機能改善）

新機能追加ではなく、設計思想と実装フローのズレの是正、旧コード整理、再発防止を目的とした改修です。

- **カード仕分けを主フローから除外**：標準版・がっつり版・引き継ぎ分析のいずれの `phases()` 配列からも `'cardSort'` を削除し、標準版は「アンカー→4択MaxDiff/Best-Worst→接戦ペアワイズ→避けたい要素」、がっつり版はこれに「トレードオフ比較→内省質問」を加えた構成に統一。カード仕分け専用画面 `cardSortScreen()` と、`startPhase()`／`goBack()` 内の対応する分岐は到達不能となったため削除。ただし `cardSortCandidates()`（MaxDiff候補プール選定に使用）と `S.cardSortLogs` のスコア再計算ロジックは、v0.9.4以前に保存された履歴を継続分析する際の互換性のために維持。
- **標準版を4択MaxDiff主軸に整理**：`MODE_CONFIG` を `anchor:8 / maxDiff:18 / pairwise:6 / avoid:1`（標準版）、`anchor:12 / maxDiff:32 / pairwise:14 / tradeoff:10 / avoid:1 / reflection:5`（がっつり版）、`additionalMaxDiff:12 / closePairwise:10 / tradeoff:8 / reflection:4`（引き継ぎ分析）に再編。旧フェーズ名 `conjoint` は `tradeoff` に統一し、引き継ぎ分析からは使われていなかった `additionalAnchor` ステップを削除。
- **戻る機能を選択画面まで拡張**：既存の回答ログベースの `answerTrail` に加え、`S.navigationStack` を追加。希望キャリア形態選択画面からアンカー質問へ進む際にスクリーン復帰用のエントリを積んでおき、アンカー質問1問目（未回答で `answerTrail` が空の状態）で「戻る」を押すと、選択済み内容を保持したまま希望キャリア形態選択画面へ戻れるようにした。現在地選択・モード選択への戻るは元々ハードコードされた遷移で正しく機能していたため変更なし。戻る処理はスコアを直接減算せず、対象ログを削除したうえで `recalculateAllScores()` により再計算する既存方針を維持。
- **旧コード・到達不能コードの整理**：現行フローから到達不可能と確認した `hybrid()` / `hybridOrCareer()` を削除。`names` / `intros` / `bucketFor()` に残っていた `cardSort` / `additionalAnchor` 用のエントリも合わせて整理し、`intros` のステップ番号をカード仕分け除去後の並びに合わせて振り直した。
- **`tieLogs` 再発防止**：`S.logs` に全回答ログを一本化する方針をコードコメントとして明記し、`tieLogs` を含む廃止済みログ配列への参照が残っていないことを再確認済み。
- **`desired_career_modes` 互換性の再確認**：`desiredModesOf()` が新形式（配列）・旧 `career_mode`（文字列／オブジェクト）のどちらでも読み替えられること、`career_mode.id` への直接参照が `branchOf()` 内の1箇所のみで null-safe に保たれていること、候補除外ではなくMaxDiff配合調整にのみ使われていることを再確認。
- **履歴データの堅牢化**：`history()` が配列以外や `id` を持たないレコード（`null`・文字列・空オブジェクトなど）を除外するフィルタを追加し、`showHistory()` 側の `user_status` 参照もオプショナルチェーンでフォールバック（`不明`）するよう防御化。破損データが混在していてもアプリ本体・履歴画面ともにクラッシュしないことを確認。

大規模なリファクタは行わず、既存のスコア計算方式（`S.logs` を正としたログ再計算）・保存形式・AI分析用プロンプト構成は維持しています。
